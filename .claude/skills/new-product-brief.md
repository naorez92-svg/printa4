# סקיל: ישיבת ייסוד לפני בנייה של מוצר חדש

## מתי להפעיל
כשהמשתמש אומר "אני רוצה לבנות מוצר חדש", "בוא נתחיל אפליקציה", "יש לי רעיון ל-SaaS" — לפני שנכתב שורת קוד אחת.

## מטרת הסקיל
לנהל ישיבת ייסוד מובנית שמסתיימת בקובץ `PRODUCT_BRIEF.md` אחד שמונע 80% מהעבודה הכפולה שקורית בפיתוח ללא תכנון.

---

## מה למדנו מהשטח (לקחים מ-printa4/בשבילי)

**כפילות לוגיקה עסקית** — מכסת הניסיון החינמי הייתה מוגדרת ב-3 מקומות: `useProfile.js`, ה-Edge Function וה-DB trigger. כשעדכנו אחד, השניים האחרים נשארו לא מסונכרנים. תיקון לקח 3 מיגרציות נפרדות.

**מסר שיווקי הוגדר אחרי** — "חסכי 3 שעות הכנה בשבוע" היה בדף הנחיתה, אבל הטקסטים בתוך האפליקציה לא חיזרו עליו. נדרש מחזור שלם של עריכה כדי לחבר את השניים.

**מינוח לא הוגדר** — "מסיון" (מילה שאינה קיימת) הגיע לפרודקשן כי לא הייתה רשימת מינוח מאושרת. תוקן רק כשמשתמש ראה את זה בפועל.

**מודל אבטחה הוגדר בדיעבד** — CORS, XSS sanitization, rate limiting ו-quota gaming התגלו בסקירות אבטחה שנעשו אחרי שהקוד כבר היה כתוב. כל אחד דרש refactor.

**סכמת DB לא תוכננה עד הסוף** — עמודת `total_booklets_created` נוספה במיגרציה 0019 כי לא חשבנו מראש שמשתמשים ימחקו חוברות כדי לאפס מכסה.

## הוראות לסוכן

### שלב א — פתח בהצהרה

```
לפני שנכתב שורת קוד — בוא נעשה ישיבת ייסוד. 
אני אשאל אותך ~30 שאלות ב-8 נושאים.
התשובות יהפכו לקובץ PRODUCT_BRIEF.md שיחסוך לנו שבועות של עבודה כפולה.
מוכן?
```

### שלב ב — 8 ערוצי שאלות (שאל בסדר, אל תדלג)

---

#### ערוץ 1: ערך ומשתמשים

**שאל:**

1. מה הבעיה המדויקת שהמוצר פותר? (משפט אחד, לא "עוזר ל-X לעשות Y" — אלא "כרגע X מבזבז/סובל מ-Z")

2. מי הוא המשתמש הראשי שמשלם? (לא "מורות ורופאים" — תאר את האדם הספציפי: גיל, מקצוע, עד כמה הוא טכני, מה כואב לו הכי הרבה)

3. מה המשתמש עושה היום בלי המוצר שלך? כמה זמן/כסף זה עולה לו?

4. מה ה-One-liner שיגרום ל-10 אנשים לתת לך את המייל שלהם?
   ← *זה המסר השיווקי המרכזי. כל טקסט אחר באפליקציה יחזק אותו.*

5. מה ה-"אה-הא מומנט" — הרגע שמשתמש חדש מרגיש לראשונה שהמוצר שווה? מתי זה קורה בדרך כלל?

---

#### ערוץ 2: מודל עסקי ומכסות

6. האם יש טיר חינמי? מה ה-hook שלו (מספיק כדי לתת ערך, לא מספיק כדי לא לשלם)?

7. מה מגבלות כל תוכנית? (מספרים מדויקים — לא "כמה שצריך")
   ← *אלה יוגדרו פעם אחת ב-constants.ts ויצוטטו משם בכל מקום.*

8. מה קורה אם משתמש מנסה לעקוף את המכסה? (למשל: מוחק ויוצר מחדש, פותח חשבון חדש, API abuse)
   ← *חשוב על זה עכשיו — לא כשיש 1000 משתמשים.*

9. איך משלמים? (Stripe, Bit, PayPal, WhatsApp manual?) מה ה-friction שמוכנים לקבל?

10. מה ה-upgrade trigger — הרגע שמשתמש צריך לשמוע "שדרג"? מה הטקסט המדויק שיגרום לו לשלם?

---

#### ערוץ 3: ארכיטקטורה טכנית

11. Frontend: React/Vue/Next/Svelte? למה? האם SSR נדרש (SEO?)

12. Backend: Edge Functions / Node / Python / serverless? איפה הלוגיקה העסקית תגור?
    ← *חוק: לוגיקה עסקית (מכסות, תשלום, הרשאות) תמיד בצד שרת + DB, לעולם לא client-only.*

13. DB: Postgres / Mongo / Firebase? מי מנהל? (Supabase, PlanetScale, Railway?)

14. Auth: magic link / Google OAuth / password? מה ה-session strategy?

15. AI/LLM (אם רלוונטי):
    - איזה מודל? (claude-opus-4-8 / gpt-4o / אחר)
    - streaming או sync?
    - output format (HTML? JSON? Markdown?)
    - איפה ה-prompt גר? (Edge Function בלבד — לעולם לא client!)
    - איך מונעים prompt injection?

16. Storage: קבצים / תמונות / PDFs — איפה? (Supabase Storage, S3, Cloudflare R2?)

17. Real-time: יש צורך ב-WebSocket / SSE / polling? לאיזה feature?

---

#### ערוץ 4: סכמת DB (לפני שנכתב מיגרציה)

18. רשום את כל הטבלאות שתצטרך. לכל טבלה: שם + כל העמודות + טיפוס + nullable? + default?
    ← *כלל: כל עמודה שתצטרך בחודש הראשון — הוסף עכשיו. מיגרציות הן friction.*

19. אילו fields הם "lifetime counters" (לעולם לא מופחתים, גם אם רשומות נמחקות)?
    ← *דוגמה: total_booklets_created — לא COUNT(booklets) אלא counter שרק עולה.*

20. אילו שאילתות תרוץ הכי הרבה? האם יש indexes מתאימים?

21. מה ה-RLS (Row Level Security) policy? (כל user רואה רק את שלו? יש admins?)

---

#### ערוץ 5: אבטחה (לפני שורת קוד)

22. מה הסודות (secrets) במוצר? למלא:
    ```
    SECRET_NAME → איפה גר (env var בצד שרת בלבד / Supabase secrets / GitHub Secrets)
    לעולם לא: VITE_ prefix לסודות, hardcode בקוד, commit לrepo
    ```

23. CORS: אילו origins מורשים? (production domain + localhost + Vercel previews?)
    ← *אל תשכח: Vercel preview URLs הן דינמיות — צריך regex.*

24. XSS surface: האם המוצר מציג HTML שנוצר ע"י AI / משתמש? אם כן — sanitization חובה לפני render.

25. Rate limiting: כמה requests לדקה/שעה per user? מה קורה כשחורגים?

26. Input validation: אילו שדות מגיעים מהמשתמש? מה ה-max length? מה מסוכן?

---

#### ערוץ 6: עיצוב ותוכן

27. Design tokens: מלא:
    ```
    primary color: #___  (שם: ___)
    secondary: #___  (שם: ___)
    background: #___  (שם: ___)
    text: #___  (שם: ___)
    success: #___  (שם: ___)
    ```
    ← *אלה יוגדרו פעם אחת ב-tailwind.config.js ולא יהיו hard-coded בשום מקום.*

28. שפה ראשית: עברית / אנגלית / דו-לשוני?
    - אם עברית: RTL בכל המסמך? (dir="rtl" על body)
    - מי כותב את הטקסטים? AI, מישהו ספציפי?

29. **רשימת מינוח** — 10 מושגים מרכזיים שחוזרים במוצר. כתוב כל אחד בצורה הסופית שלו:
    *(דוגמה: "חוברת עבודה" ולא "דף תרגיל" ולא "worksheet" — אחיד בכל מקום)*

30. אילו empty states / error messages יש? מה הטון? (מקצועי, ידידותי, מצחיק?)

31. מה ה-loading state experience? (spinner, skeleton, progress bar, הודעות מתחלפות?)

---

#### ערוץ 7: deployment ו-infra

32. Hosting frontend: Vercel / Netlify / Cloudflare Pages / אחר?

33. אילו deployments הם אוטומטיים ואילו ידניים?
    ```
    push to main → [מה קורה אוטומטית?]
    DB migrations → [ידני? אוטומטי? מי מפעיל?]
    Edge Functions → [ידני? אוטומטי?]
    ```

34. מה ה-branch strategy? (main = production, dev branch, feature branches?)

35. Analytics: האם צריך event tracking? (PostHog, Mixpanel, Plausible?) אילו events חשובים?

36. Monitoring: איך יודעים שמשהו נשבר? (Sentry? Supabase logs? alerts?)

---

#### ערוץ 8: scope ו-MVP

37. מה **בתוך** ה-MVP? (רשימה של features שחייבים להיות ביום אחד)

38. מה **מחוץ** ל-MVP ונרשם במפורש כ-"לא עכשיו"?
    ← *זה קריטי — כל feature שלא נרשם כ-"לא עכשיו" יגיע כ-"למה עוד לא?"*

39. מה מדד ההצלחה של ה-MVP? (X משתמשים? Y הכנסה? Z conversions?)

40. מה הסיכון הגדול ביותר שיכול להרוג את המוצר? (טכני? שיווקי? משפטי?)

### שלב ג — תפוקת הסקיל

אחרי שאספת את כל התשובות, צור קובץ `PRODUCT_BRIEF.md` עם המבנה הבא:

```markdown

## 1. One-liner
[המשפט שיגרום למשתמש לתת מייל]

## 2. משתמש ראשי
[תיאור + כאב + נתיב שימוש]

## 3. מסר שיווקי (להשתמש בו בכל מקום!)
- כותרת: "..."
- תת-כותרת: "..."
- מספרים מרכזיים: "X [יחידה] ב-Y [זמן]"
← כל טקסט באפליקציה שנוגע ל-value prop חייב להשתמש במשפטים האלה

## 4. מודל עסקי
| תוכנית | מחיר | מגבלה | כפיית המגבלה |
|--------|------|--------|--------------|
| חינמי  | ₪0   | X lifetime | DB trigger |
| ...    | ...  | ...    | ...          |

FREE_LIMIT = X  ← ערך יחיד, מוגדר ב-constants.ts, מצוטט משם בכל מקום

## 5. סכמת DB
[כל הטבלאות עם כל העמודות — לפני שנכתב מיגרציה]

lifetime counters (לעולם לא מופחתים):
- [שם_עמודה] ב-[טבלה]

## 6. ארכיטקטורה
Frontend: ...

Auth: ...

## 7. מיפוי סודות

מורשים: production domain, localhost:5173, /^https:\/\/[repo]-git-[^.]+\.vercel\.app$/

## 10. מינוח מאושר

מדד הצלחה: [מספר ספציפי]

## 12. סיכוני MVP

## הערה לסוכן

אל תדלג על שאלות. "נחליט אחר כך" הוא התשובה שגורמת ל-refactor בשבוע 4.
אם התשובה לשאלה היא "לא יודע" — זה מידע חשוב, רשום אותו ב-BRIEF עם "TBD" ותאריך לכיוון.
הישיבה הזאת אמורה לקחת 30-60 דקות. כל שעה שמשקיעים כאן חוסכת 10 שעות של עבודה כפולה אחר כך.
