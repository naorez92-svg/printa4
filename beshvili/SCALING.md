# 🗺️ מפת סקייל — בשבילי

מסמך החלטה: מה נשבר, באיזה גודל, כמה זה עולה, ובאיזה סדר לתקן.
לא קוד — רק המפה. עדכון אחרון: יוני 2026.

---

## TL;DR

- **התשתית המנוהלת (Vercel + Supabase + Auth) תגיע למיליון** כמעט בלי מאמץ.
- **התקרה האמיתית היא Anthropic** — לא טכנית אלא **עלות**: מודל ה-3-חוברות-חינם הופך יקר מאוד בקנה מידה.
- **בקוד שלנו יש 2 "פצצות זיכרון"** (`admin-stats`, `send-followup`) שיישברו כבר ב-1k–10k משתמשים — אלה התיקונים הדחופים.
- אנחנו היום על **~100 משתמשים**. רוב הדברים פה הם לבנות *נכון עכשיו* כדי שלא יכאב אחר כך — לא לבנות מיליון היום.

## 💰 התקרה האמיתית: עלות Anthropic

מודל: `claude-sonnet-4-6` · input $3/1M · output $15/1M · prompt caching על system.

חוברת ממוצעת ≈ **20k–30k טוקני פלט** → **~$0.25–0.40 לחוברת** (ה-input מוזל ב-caching).

**מה זה אומר בקנה מידה (free tier = 3 חוברות/משתמש):**

| משתמשים | אם 30% מפעילים ויוצרים 3 | עלות API משוערת |
|---|---|---|
| 1,000 | 900 חוברות | ~$300 |
| 10,000 | 9,000 | ~$3,000 |
| 100,000 | 90,000 | ~$30,000 |
| 1,000,000 | 900,000 | **~$300,000** |

👉 **המסקנה האסטרטגית:** ה-free tier הוא בור העלות. במיליון משתמשים, או שההמרה לתשלום מכסה את זה, או שצריך לשנות את מבנה ה-free (פחות חוברות חינם / מודל איטי יותר ל-free / caching של חוברות נפוצות). **זו החלטה עסקית, לא הנדסית** — וכדאי להחליט עליה הרבה לפני מיליון.

מה שכבר עוזר: prompt caching (מופעל), retry על עומס (הוספנו). מה שיעזור בקנה מידה: tier גבוה מול Anthropic, תור/throttling, ו-caching של חוברות חוזרות.

## 🔴 פצצות הזיכרון בקוד (יישברו מוקדם)

### 1. `admin-stats` — נשבר כבר ב-1,000 משתמשים
- `listUsers({ perPage: 1000 })` → **מעל 1,000 משתמשים הדשבורד סופר חלקית בלבד.**
- טוען את *כל* החוברות, *כל* הפרופילים, ועד 100k events **לזיכרון ה-JS** ומחשב שם.
- **התיקון:** ספירה ואגרגציה ב-SQL (`count`, `group by`, פאגינציה) במקום ב-JS. נשבר: ~1k. מאמץ: בינוני.

### 2. `send-followup` / `send-renewal-reminder` — נשבר ב-10k–50k
- טוענים את *כל* הפרופילים + *כל* החוברות + `listUsers(1000)` לזיכרון, בלולאה.
- **התיקון:** עיבוד ב-batches עם פאגינציה / תור. נשבר: ~10k. מאמץ: בינוני.

### 3. טבלת `events` — גדלה בלי הגבלה
- כל אירוע שימוש נשמר; האנליטיקס טוענת עד 100k שורות ל-JS.
- **התיקון:** retention (purge מעל 30–90 יום — כבר מוכן ב-`pg_cron`, צריך להפעיל) + אגרגציה ב-DB / materialized views. נשבר: ~50k. מאמץ: בינוני.

### 4. אינדקסים
- רוב הנתיבים החמים **כבר מאונדקסים** (`booklets(user_id, created_at)`, `booklets(share_token)`, `events(event/user/anon/session, created_at)`). מצב טוב.
- פערים שנותרו (טופלו במיגרציה `0030`): ספירות תאריך גלובליות על `booklets(created_at)`, `children(user_id)`, ו-`profiles(pro_since)` לחידושים.
- **מאמץ: נמוך — PURE WIN, אפס סיכון.** ✅ בוצע.

## 🛣️ הסדר המומלץ

### שלב 1 — עכשיו (נכון, זול, סיכון נמוך)
1. **אינדקסים** (PURE WIN, אפס סיכון) — להתחיל מפה.
2. **`_shared` (SSOT)** — getCors, constants, anthropic, email.
3. **תיקון פצצות הזיכרון** — `admin-stats` ו-`send-followup` לאגרגציה ב-DB.
4. **הפעלת purge ל-events** (pg_cron).

### שלב 2 — ב-10k+
- analytics בצד DB (materialized views), retention מלא, בדיקת connection pooling (pgbouncer).

### שלב 3 — ב-100k+
- אסטרטגיית rate-limit/עלות מול Anthropic (tier, תור, caching), monitoring/alerting, אולי read replica.
- **החלטה עסקית על מבנה ה-free tier** (התקרה האמיתית).
